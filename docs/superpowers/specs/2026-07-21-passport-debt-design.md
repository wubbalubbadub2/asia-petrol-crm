## Цель

Добавить третий вариант экспорта паспорта — **«Паспорт (долги)»** (сейчас 3-я опция дропдауна, disabled). Формат = существующий **detail-экспорт** (главная строка сделки + под-строки по отгрузкам) + **6 колонок отсрочки/плановых дат оплаты**. Для этого нужны новые поля «условий оплаты» на сделке и их ввод в UI.

## Ответы клиента (решения)

1. **Строка сделки остаётся** (коммент в шаблоне «не нужна» — игнорировать). Агрегаты (Заявлено/Остаток/Оплата/Баланс/Долг) — на главной строке, как в detail.
2. **Отсрочка = 2 набора полей на сделку** (продавец / покупатель). «Приложение как сущность» — отдельная будущая задача, НЕ здесь.
3. **Режим отсрочки** на сторону: `shipment` («с даты отгрузки», система считает) или `other` («прочее», ручная дата + текст-заметка).
4. **Прод.** = продавец = наш поставщик; **Покуп.** = покупатель.
5. Режим `other` → **ручное поле плановой даты** (менеджер вписывает), идёт в отчёт.
6. **Просрочка** — подсветка красным просроченных плановых дат **прямо в debt-экспорте**.

## Разбор шаблона (колонки 56–61)

Колонки 1–55 = как в detail-экспорте (`passport-detail-excel.ts`). Новые:

| # | Заголовок | Значение |
|---|---|---|
| 56 | Отсрочка платежа Прод., дн. | `deals.supplier_deferral_days` |
| 57 | Дата начала отсрочки (Прод.) | режим: «с даты отгрузки» / текст-заметка (при `other`) |
| 58 | Плановая дата оплаты Прод. | см. формулу ниже; **красным при просрочке** |
| 59 | Отсрочка платежа Покуп., дн. | `deals.buyer_deferral_days` |
| 60 | Дата начала отсрочки (Покуп.) | режим / текст-заметка |
| 61 | Плановая дата оплаты Покуп. | формула ниже; **красным при просрочке** |

В шаблоне: под-строки-отгрузки, дни 30/45, режим «с даты отгрузки», плановая = дата СНТ + дни (Прод. по вход. СНТ, Покуп. по исход. СНТ).

## Компонент 1 — схема (миграция)

На `deals` добавить по 4 колонки на сторону:

```sql
ALTER TABLE deals
  ADD COLUMN supplier_deferral_days   INT,
  ADD COLUMN supplier_deferral_mode   TEXT CHECK (supplier_deferral_mode IN ('shipment','other')),
  ADD COLUMN supplier_deferral_note   TEXT,   -- текст «прочее» (col 57)
  ADD COLUMN supplier_planned_pay_date DATE,  -- ручная плановая дата при mode='other'
  ADD COLUMN buyer_deferral_days      INT,
  ADD COLUMN buyer_deferral_mode      TEXT CHECK (buyer_deferral_mode IN ('shipment','other')),
  ADD COLUMN buyer_deferral_note      TEXT,
  ADD COLUMN buyer_planned_pay_date   DATE;
```

Все nullable (NULL = условия не заведены → пусто в отчёте). Никаких триггеров — плановая дата считается в экспортере, не хранится.

## Компонент 2 — плановая дата (формула, в экспортере)

На каждой **под-строке** (отгрузке) для каждой стороны:

- **mode = 'shipment'**: планов. дата = дата СНТ этой стороны + дни.
  - Прод.: `loading_date` (вход. СНТ) + `supplier_deferral_days`.
  - Покуп.: `date` (исход. СНТ) + `buyer_deferral_days`.
  - Если нужной даты СНТ или дней нет → пусто.
- **mode = 'other'**: планов. дата = `*_planned_pay_date` (ручная, на уровне сделки — одна на все под-строки), колонка режима (57/60) показывает `*_deferral_note`.
- **mode = NULL**: обе колонки пусто.

Колонка «дни» (56/59) и «режим/заметка» (57/60) — на под-строках (значения с уровня сделки, повторяются). На главной строке сделки колонки 56–61 пустые (как в шаблоне).

## Компонент 3 — просрочка (красный в Excel)

Ячейка плановой даты (58/61) выделяется **красным шрифтом** (ExcelJS `font.color`), если:
- планов. дата **< сегодня** (дата запуска экспорта), И
- сторона **не закрыта**: Прод. → `deals.supplier_balance` указывает на непогашенный остаток; Покуп. → `deals.buyer_debt > 0`.

> На этапе плана уточнить знак `supplier_balance` (проверить против rollup-конвенции проекта), чтобы «мы ещё должны поставщику» определялось верно. Долг покупателя `buyer_debt > 0` = покупатель ещё должен нам.

Просрочка — на уровне под-строки (её планов. дата) × непогашенность стороны на уровне сделки (оплаты хранятся по сделке, не по отгрузке — это осознанное упрощение).

## Компонент 4 — ввод в UI (страница сделки)

Новый блок **«Условия оплаты»** на `deals/[id]/page.tsx`. По стороне (Поставщик / Покупатель):
- `дни` — числовое поле.
- `режим` — select: «с даты отгрузки» (`shipment`) / «прочее» (`other`).
- при `other` дополнительно: `заметка` (текст) + `плановая дата` (date).

Оптимистичный ввод — паттерн проекта (`updateDeal` / `applyDealPatch`, мгновенно, откат+toast на ошибке). Поля добавить в `Deal` type + `LIST_SELECT` (`use-deals.ts`).
