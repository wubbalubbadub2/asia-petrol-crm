# Отчёт «Сбор по валюте» — дизайн

**Дата:** 2026-07-22
**Источник:** `files/Обработка сбор по валюте (1).docx`
**Предшественник:** `2026-07-20-fx-conversion-reports-design.md` (помесячные отчёты)

## Контекст

Клиент прислал вторую редакцию ТЗ. Текст правил конвертации **не изменился ни на
строку** — дописан только хвост, который меняет форму витрины:

- 25 обязательных колонок отчёта (строка = сделка, а не месяц);
- из строки можно провалиться в сделку / в основной паспорт;
- выгрузка в Excel «идентична выгрузке паспорта в формате Паспорт Детальный,
  с датами и суммами в разрезе»;
- «фильтрация должна быть один в один как в паспорте».

Первая строка ТЗ: «Весь анализ делится на два отчёта (сбор и анализ данных)».
Уже сделанные помесячные отчёты = «анализ», новая таблица = «сбор».

Вся инфраструктура конвертации существует и работает: таблица `fx_rates`,
суточная загрузка НБ РК / НБ КР (`/api/cron/fx-rates`), функции `fx_rate`,
`fx_rate_month`, `fx_convert`, `fx_convert_month`, пивот через USD.
Меняется только витрина.

## Принятые решения

| # | Решение | Обоснование |
|---|---|---|
| 1 | Помесячные отчёты остаются | Клиент сам делит анализ на два отчёта |
| 2 | Конвертация **на клиенте**, в TS | Формулы в одном месте, переиспользуются фильтры/таблица/экспорт, доступна самопроверка на паритет с паспортом |
| 3 | Фильтры **копируются**, `/deals` не трогаем | Осознанный дубль: рабочая страница сделок не переписывается. Копия живёт отдельным файлом |
| 4 | Вкладки `KG / KZ / Все` + тумблер `₸ / $` | В единой валюте свод по KG+KZ впервые имеет смысл — ради этого отчёт и делается |
| 5 | Баланс и Долг — **формула паспорта**, не своя | Отчёт ничего не пересматривает, только пересчитывает по курсу |

## Правило курса

Уточнение клиента 2026-07-22:

> «Если мы показываем данные на сегодня, то мы должны взять курс вчерашнего дня —
> сегодня курс ещё не зафиксировался и в течение дня может меняться. Для всех дат
> начиная со вчера и ранее курс уже зафиксирован, берём курс того дня.»

Формально: `эффективная дата курса = min(дата события, вчера)`.
Дальше — как сейчас: берётся последний курс с `date <= эффективная дата`
(выходные и праздники автоматически подтягивают последний рабочий день).

Правило применяется в ДВУХ местах, чтобы оба отчёта считали одинаково:
- TS-функция конвертации (новый отчёт);
- SQL `fx_rate()` — миграция, клампим `p_date` до `current_date - 1`.

Загрузку курсов не меняем: сегодняшний курс продолжаем сохранять, кламп
происходит на чтении (обратимо, историю не портит).

## Источники данных

Всё уже умеет тянуть экспортёр «Паспорт Детальный» — выносим в общий модуль
`src/lib/data/deal-events.ts`, чтобы отчёт и выгрузка ели из одной тарелки:

| Что | Откуда | Объём за год |
|---|---|---|
| Сделки | `useDeals({ year, isArchived: false })` — как в паспорте | ~400 |
| Отгрузки | `fetchShipmentsByDeals` (чанки по 150 id, `fetchAllPaginated`) | ~7 000 |
| Оплаты | `fetchPaymentsByDeals` (та же схема) | ~2 000 |
| Курсы | `fx_rates` за диапазон дат выборки + месяцы | ~500 |

Пагинация обязательна: PostgREST режет на 1000 строк молча (наступали дважды).

## Конвертация: что из чего

Хранимые итоги сделки сконвертировать нельзя — у них нет даты. Каждая денежная
колонка пересобирается из событий, каждое событие берёт курс своей даты.

| Колонка | Формула | Дата курса |
|---|---|---|
| Приход сумма | `Σ (цена пост. × loading_volume)` по строкам реестра | `loading_date` (вход. СНТ) |
| Приход объем | `Σ loading_volume` | — (объём не конвертируется) |
| Оплата (пост.) | `Σ deal_payments[side=supplier].amount` | `payment_date` каждой строки |
| Отгружено сумма | `Σ (цена покуп. × shipment_volume)` | `date` (исход. СНТ) |
| Отгружено тонн | `Σ shipment_volume` | — |
| Оплата (покуп.) | `Σ deal_payments[side=buyer].amount` | `payment_date` |
| Сумма (жд) | `Σ shipped_tonnage_amount` | `loading_date` — по ТЗ «оплата экспедитору берётся по дате входящего СНТ» |
| Сумма грузоотправления | `Σ additional_expenses` | `loading_date` |
| Факт объем | KZ → `Σ loading_volume`, KG → `Σ shipment_volume` | — |
| **Цена** (пост.) | `Приход сумма ÷ Приход объем` | производная |
| **Цена** (покуп.) | `Отгружено сумма ÷ Отгружено тонн` | производная |
| **Тариф факт** | `Сумма ÷ Факт объем` | производная |
| **Баланс** | `Приход сумма − Оплата пост. + жд + грузоотправитель` | слагаемые уже сконвертированы |
| **Долг** | `Оплата покуп. − Отгружено сумма` | то же |

Цена как производная — прямо из ТЗ: «конвертированная сумма делится на
отгруженный объём», с примером `219 745,4 / 646,310 = 340 $`.

**Нет даты у события** → среднемесячный курс месяца сделки (`=СРЗНАЧ()` в
терминах ТЗ, `fx_convert_month` в коде). Если нет и месяца — ячейка пустая,
строка помечается как неполная.

**Баланс и Долг** повторяют формулу БД (`compute_deal_derived_fields`, 00112)
слово в слово, включая условия по галочкам «жд в цене» / «грузоотправитель в
цене». Условие «валюта сделки == валюта логистики» вычисляется по **исходным**
валютам, а не по целевой — иначе после конвертации оно выполнялось бы всегда и
баланс разошёлся бы с паспортом по составу, а не только по курсу.

**Объёмы, тоннаж и количество вагонов не конвертируются никогда.**

### Исходная валюта суммы

Валюта у сделки не одна. Каждая сумма конвертируется ИЗ своей валюты:

| Сумма | Исходная валюта |
|---|---|
| Приход сумма, оплаты поставщику, баланс | `deals.supplier_currency` |
| Отгружено сумма, оплаты покупателя, долг | `deals.buyer_currency` |
| Сумма жд, сумма грузоотправителя, тариф | `shipment_registry.currency ?? deals.logistics_currency` |
| Отдельная оплата | `deal_payments.currency ?? валюта своей стороны` |

Правило «строка реестра важнее сделки» повторяет уже принятое в реестре и
ДТ-КТ. Цепочка KGS → USD → KZT (НБ КР, затем НБ РК) отрабатывает
существующим пивотом через USD.

## Интерфейс

Страница `/reports/collection`, пункт меню «Сбор по валюте» в разделе «Отчёты».
Существующий `/reports` переименовывается в «Анализ по валюте».

Шапка: вкладки `KG (Экспорт) / KZ (Внутренний) / Все`, тумблер валюты `₸ / $`,
год, копия фильтров паспорта, кнопка Excel.

Провал в данные: № сделки — ссылка на карточку сделки; отдельная иконка
«открыть в паспорте» ведёт на `/deals` с тем же фильтром.

Группы компаний: в интерфейсе — одна ячейка с цепочкой; в Excel — отдельные
колонки Группа 1 / Группа 2 / Группа 3 (уточнение клиента 2026-07-22).

## Excel

Выгрузка — «Паспорт Детальный» один в один: главная строка сделки плюс
под-строки по каждой отгрузке с датами. Единственное отличие — все денежные
значения в выбранной валюте, каждое по курсу своей даты. Новый формат не
рисуем: переиспользуем `passport-detail-excel.ts`, добавив в него режим валюты.

Под-строки конвертируются по собственным датам строки (вход. СНТ для сумм
поставщика и логистики, исход. СНТ для сумм покупателя, дата платежа для оплат).

## Проверка

1. **Паритет с паспортом (главный тест).** Если целевая валюта совпадает с
   валютой сделки, пересчитанные значения обязаны совпасть с хранимыми:
   `supplier_shipped_amount`, `supplier_payment`, `supplier_balance`,
   `buyer_shipped_amount`, `buyer_payment`, `buyer_debt`, `invoice_amount`,
   `additional_expenses_amount`. Юнит-тест на фикстурах + живая сверка на проде.
2. **Курс.** Событие сегодняшней датой берёт вчерашний курс; вчерашнее и более
   раннее — курс своего дня; выходные подтягивают последний рабочий день.
3. **Полнота.** Число строк отчёта == числу строк паспорта при тех же фильтрах.
4. **Excel.** Открыть выгрузку, сверить суммы под-строк с экраном и с паспортом.

## Границы
